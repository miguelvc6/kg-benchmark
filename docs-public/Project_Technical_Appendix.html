Frozen World-State Context

A world-state entry is the frozen graph context attached to one historical repair. It is keyed by the benchmark case id and contains four layers plus optional T-box change context.

L1

Ego node

The focus entity id, label, description, sitelink count, compact property map, and optional popularity block.

L2

Labels

A deterministic id-indexed map of labels and descriptions for the focus node, target property, neighbors, and constraint-related ids.

L3

Neighborhood

A bounded one-hop outgoing neighborhood with property ids, target ids, target labels, and target descriptions.

L4

Constraints

The current property constraint inventory, including constraint-family ids, qualifier details, and human-readable rule summaries.

This example is deliberately compact. Real entries may contain more properties, labels, neighborhood edges, and constraints.

Important temporal rule: the frozen world state may describe the contemporary graph, but the target property's historical pre-repair state is reconstructed from repair metadata during prompting and evaluation. This prevents post-repair target values from leaking into model-visible inputs.

Popularity Metadata

Popularity is a controlled covariate. It helps analyze whether models behave differently on widely visible entities and long-tail entities.

Raw components

For each focus entity, the pipeline collects three signals: English Wikipedia pageviews over a 365-day window, graph out-degree, and total sitelinks count.

Log scaling

Each raw component is transformed with log1p(value) before ranking, reducing the dominance of extremely popular entities.

Percentile normalization

Each transformed component is converted to a percentile rank within the current focus-entity set. Ties receive the average percentile of their tied range.

Composite score

The final score is a weighted sum: pageviews percentile contributes 50%, out-degree percentile contributes 30%, and sitelinks percentile contributes 20%.

Formula: popularity_score = 0.5 * pageviews_norm + 0.3 * degree_norm + 0.2 * sitelinks_norm 
{
  "popularity": {
    "score": 0.553479,
    "components": {
      "pageviews_365d": 0,
      "out_degree": 19,
      "sitelinks": 2
    },
    "normalization": {
      "pageviews_norm": 0.338707,
      "degree_norm": 0.853548,
      "sitelinks_norm": 0.640305
    },
    "policy": {
      "window_days": 365,
      "wiki": "enwiki",
      "pageviews_project": "en.wikipedia.org"
    }
  }
}

Popularity buckets used in summaries are derived by sorting cases by score: the bottom 20% are tail, the top 20% are head, and the rest are mid. Missing scores are bucketed as unknown.

How Cases Are Classified

Classification is deterministic for fixed artifacts. It first decides the repair locus, then decides what information source is required for A-box cases.

What the classifier is trying to answer

The classifier is not trying to predict whether a repair is correct. The historical repair target already defines the benchmark target. Classification instead answers a methodological question: what kind of information would a system need in order to reproduce or justify that target repair?

This distinction matters because two cases can have similar surface shapes but require different capabilities. Removing a value because a format rule is violated is a different task from replacing a value using evidence in a neighboring node, and both are different from changing the schema because the property constraint itself has become outdated.

Inputs used for classification

Input Role in the decision
Historical repair record Provides the focus entity, target property, repair track, old value, new value, action, violation report context, and T-box constraint delta when present.
Frozen world-state context Provides local graph evidence, labels, descriptions, one-hop neighbors, and current property constraint metadata.
Reconstructed pre-repair target state Replaces the current target-property values with the historical pre-repair values so local evidence checks do not accidentally see the answer after the repair.
Constraint metadata Identifies rule-deterministic cases and provides the constraint-family inventory used for T-box subtype inference.
Resolved truth tokens Represents the historical repair target as comparable tokens, such as QIDs, literals, dates, or normalized values, so the classifier can ask where the truth was available.

Decision priority

The classifier runs in a fixed priority order. Earlier decisions intentionally mask later ones. For example, a delete action is treated as logical rejection before local evidence is considered; a T-box reform is kept out of the A-box Type A/B/C taxonomy because it is a schema-level repair.

Decision order Classification result Reason
World-state entry missing TypeC / EXTERNAL Without context, local reasoning cannot be safely established. This is a low-confidence fallback, not proof that the case is truly external.
Historical target is schema-level T_BOX The repair changed the property constraint signature rather than an entity statement.
A-box delete action TypeA / REJECTION Deleting an invalid value is treated as logical rejection or cleaning.
Rule-deterministic repair TypeA / LOGICAL The constraint itself uniquely identifies the repair signal.
Truth found in local context TypeB / LOCAL... The target truth appears in focus-node text, synthetic pre-repair target values, neighbor ids, or neighbor text.
No local or rule-deterministic evidence TypeC / EXTERNAL The repair requires information beyond the local graph context.

A-box logical rules

A case is Type A when the repair can be determined from the rule, the violation shape, or the rejection of an invalid value, without needing graph traversal or external evidence. The classifier distinguishes two main Type A pathways.

REJECTION

Delete-as-cleaning

If the historical A-box action deleted the violating value, the case is labeled Type A because the task is to reject a value that should not be present. The system does not need to retrieve a replacement truth.

LOGICAL

Rule-deterministic repair

If the constraint uniquely points to the fix, the case is logical. Supported first-wave signals include format constraints, singleton one-of constraints, and range constraints when the repaired value matches a rule-implied boundary.

Rule-deterministic examples

Constraint family Why it can be Type A Confidence behavior
Format A malformed literal can often be corrected by the regular expression or by the format violation itself, such as removing an illegal trailing character. Usually high when constraint metadata is present; report-type fallback can also support a format signal.
One-of singleton If exactly one value is allowed, the rule itself identifies the target value. High when the repaired truth matches the singleton allowed value; medium when singleton structure is present but truth evidence is weaker.
Range boundary If the repaired value is the rule boundary itself, the repair can be inferred from the constraint rather than external knowledge. High for direct numeric or date boundary matches; medium when the signal is only visible through a rule summary.

A-box local evidence

Local evidence is searched after reconstructing the target property in its pre-repair state. QID truths require exact id matches. Literal truths use normalized text matching. Date matches must preserve enough precision; a year-only mention does not satisfy a full-date truth.

Local evidence is gathered into separate buckets. The classifier then checks whether the historical repair target appears in any bucket. The subtype records where the evidence was found.

Local subtype Evidence source Interpretation
LOCAL_FOCUS_PREREPAIR_PROPERTY Values already present on the reconstructed focus node before the target repair. The graph already contains the needed truth near the target property; the task is selecting or preserving the right local value.
LOCAL_NEIGHBOR_IDS Exact ids in one-hop outgoing neighbor targets. The answer is structurally visible in the immediate graph neighborhood.
LOCAL_TEXT Focus-node or neighbor labels and descriptions. The answer is visible in local text, but this is slightly weaker evidence than an exact graph-id match.
LOCAL_MIXED More than one local source type. The repair target is locally available through multiple evidence channels.

How truth matching works

The classifier first extracts comparable truth tokens from the historical target. For A-box updates, this usually means the repaired new value; for deletes, no replacement truth is expected. Values are normalized into comparable atoms so that graph ids, literals, and dates can be matched consistently.

QID repairs require exact QID matches. Literal repairs are normalized before comparison so superficial casing or spacing differences do not dominate. Date repairs require adequate precision: a local mention of only a year is not treated as enough evidence for a full date repair. Post-repair target-property values from the frozen world state are excluded so the classifier does not label a case local merely because the answer is already present after the historical repair.

External fallback

A case becomes Type C when the repair target is not rule-deterministic and the truth cannot be found in the local buckets. This does not mean the repair is unknowable. It means a benchmarked system would need some non-local source: an external database, source citation, web evidence, longer graph traversal, or domain knowledge not present in the supplied local context.

Confidence is medium by default for external fallbacks. It drops to low when the case should have truth tokens but the available artifacts cannot extract them cleanly, because then the external label is partly a consequence of missing or weak metadata.

T-box subtypes

T-box cases are assigned schema-change subtypes such as range widened, range narrowed, allowed set expanded, allowed set contracted, generic schema update, or coincidental schema change. Clear directional changes receive higher confidence than weakly causal or coincidental changes.

T-box subtype What changed Typical confidence
RELAXATION_RANGE_WIDENED A numeric or date range became more permissive, such as a lower minimum or higher maximum. High when the direction is clear.
RESTRICTION_RANGE_NARROWED A range became more restrictive, such as a higher minimum or lower maximum. High when the direction is clear.
RELAXATION_SET_EXPANSION An allowed set grew, such as more permitted values or types. High when the before/after set relation is clear.
RESTRICTION_SET_CONTRACTION An allowed set shrank. High when the before/after set relation is clear.
SCHEMA_UPDATE The constraint signature changed, but the change is not cleanly reducible to a narrower directional family. Medium by default.
COINCIDENTAL_SCHEMA_CHANGE The property schema changed near the event, but the violation type does not map convincingly to the changed constraint family. Low because causality is weak.

What the decision trace records

Each classification includes a decision trace so later audits can see why the class was assigned. The trace records whether the case was a delete, whether a rule-deterministic branch fired, whether local evidence was found, whether the classifier fell back to external, and which final branch produced the label. 

{
  "classification": {
    "class": "TypeB",
    "subtype": "LOCAL_NEIGHBOR_IDS",
    "confidence": "high",
    "decision_trace": [
      {"step": "is_delete", "result": false},
      {"step": "rule_deterministic", "result": false},
      {
        "step": "local_availability",
        "result": true,
        "evidence": {
          "matches": [
            {"source": "NEIGHBOR_IDS", "token": "Q123"}
          ]
        }
      },

      {"step": "branch", "result": "local_match"}
    ],
    "rationale": "Truth tokens matched neighbor identifiers."
  }
}

How to read confidence

Confidence is a classifier confidence, not a model-performance score and not a truth probability. High means the artifact evidence cleanly supports the label. Medium usually means the branch is plausible but less direct, such as local text evidence or a generic schema update. Low means the label is a fallback, a coincidental T-box signal, or depends on incomplete context.

Prediction And Proposal Validity

A model prediction is first normalized into a proposal contract. Evaluation then asks whether that normalized proposal is executable and successful.

A-box proposal contract

  • Targets one entity id and one property id.
  • Uses operations from SET, ADD, REMOVE, and DELETE_ALL.
  • Normalizes ids, finite numbers, dates, provenance, uncertainty, and a deterministic hash.
  • Is executable only when the proposal target matches the benchmark entity and target property.

T-box proposal contract

  • Targets one property id and one constraint-family id.
  • Uses supported schema-reform actions such as set expansion, set contraction, range widening, range narrowing, or generic schema update.
  • Normalizes the proposed post-reform constraint signature.
  • Rejects constraint-family ids outside the case-local allowlist when that allowlist is available.

Auditability requirements

A proposal can normalize successfully but still fail acceptance if it is not auditable. Acceptance requires a non-empty rationale, at least one usable provenance entry, and a numeric proposal-level uncertainty confidence.

Provenance kind Counted as complete when
WEB A URL is present.
KG A graph node id is present.
HISTORY A revision id is present.
OTHER The entry is structurally present and not one of the stricter provenance kinds above.

Exact Evaluation Metrics

Metrics are computed per case and then averaged over applicable cases. Some metrics only apply to A-box or T-box records.

A-box metrics

Metric Exact calculation
proposal_valid True when a normalized proposal object exists for the case.
proposal_executable True when the proposal target entity equals the benchmark focus entity and the proposal target property equals the benchmark target property.
a_box_exact_value_match Apply proposal operations to the reconstructed pre-repair property state; true when the resulting target-property values equal the historical repaired target values.
a_box_exact_action_match Derive the action from before/after values: delete if values disappear, create if absent values appear, update if values change, otherwise no-op. True when this derived action equals the historical repair action.
functional_success True when the proposal is executable and the resulting target-property values exactly match the historical repaired values.
exact_historical_agreement True when both exact action match and exact value match are true.
a_box_regression_pass True when the count of supported local target-property constraint violations after applying the proposal is less than or equal to the count before applying it.
information_preservation 1.0 for exact action and value agreement; -0.5 when the historical action was not delete but the proposal removes all target values; otherwise 0.0.
accepted True when functional success, regression pass, and auditability completeness are all true.

T-box metrics

Metric Exact calculation
changed_constraint_type_hit True when the proposed target constraint-family id appears in the historical set of changed constraint types. If that set is absent, it is inferred from the before/after signatures.
proposal_executable True when the proposal target property equals the benchmark target property and changed_constraint_type_hit is true.
exact_action_match True when the proposed schema action exactly equals the historical classification subtype.
exact_signature_match True when the normalized proposed post-reform signature exactly equals the normalized historical post-reform signature.
exact_historical_agreement True when exact action match and exact signature match are both true.
functional_success For T-box cases, this is intentionally strict and equals acceptance: valid, executable, exact historical agreement, and auditability complete.
semantic_success True when the proposal is executable, targets the historical constraint family, maps to the same semantic reform family as the historical change, and its signature direction is compatible when direction can be inferred.
signature_after_jaccard Jaccard overlap between the set of normalized JSON entries in the proposed post-reform signature and the historical post-reform signature.
proposal_admits_current_values For supported one-of, range, and format constraints, checks whether current target-property values would be admitted by the proposed post-reform signature. It is null when the check is not applicable.
accepted True when the proposal is valid, executable, an exact reform match, and auditability complete.

Shared metrics

Provenance completeness

Number of complete provenance entries divided by total provenance entries. Empty or malformed provenance receives 0.0.

Auditability complete

1.0 when rationale is non-empty, provenance exists with completeness above 0.0, and uncertainty has numeric confidence. Otherwise 0.0.

Conversion rate

1.0 for an accepted attempted proposal, 0.0 for an attempted but unaccepted proposal, and null when no proposal attempt exists.

Tokens to fix

If accepted and token usage exists, this is proposal total tokens plus diagnosis total tokens. It is null for unaccepted proposals or missing usage.

How Constraint Validity Is Checked

Constraint checks are deliberately conservative. The evaluator only counts supported families represented in the stored constraint metadata, rather than pretending to reproduce every Wikidata validation behavior.

Constraint family Validity check
Format constraint Extract regular-expression patterns from qualifier P1793. A violation is counted if any target value fails to fully match all available pattern alternatives.
One-of constraint Extract allowed values from qualifier P2305. A violation is counted if any target value is outside the allowed set.
Range constraint Extract numeric lower and upper bounds from range qualifiers. A violation is counted if a numeric target value falls below the minimum or above the maximum.

A-box regression rule

regression_pass = supported_violations_after <= supported_violations_before

This means a proposal is not rewarded for fixing the target value while creating additional supported target-property violations on the same focus node.

T-box admissibility rule

For supported T-box families, the evaluator can ask whether the proposed schema would admit the current target-property values. One-of reforms check membership in the proposed allowed set, range reforms check proposed numeric bounds, and format reforms check proposed regular expressions.

Aggregation And Reporting

Per-case traces are aggregated into summary rates and means. Summaries are grouped so researchers can inspect behavior by information class, subtype, track, ablation bundle, and popularity bucket.

Rates are computed as arithmetic means over cases where the metric is numeric. Metrics that do not apply to a track remain null for that track and are excluded from the denominator. Summary objects therefore include applicability counts so that, for example, T-box-only semantic metrics are not mistaken for all-case metrics.

The reporting layer also tracks proposal parse errors, proposal request errors, track-diagnosis request errors, parser-error messages, aggregate token totals, diagnosis presence, and exact track-diagnosis accuracy when diagnosis artifacts are supplied.
