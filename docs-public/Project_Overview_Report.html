WikidataRepairEval

A benchmark for studying how knowledge graphs can be repaired, verified, and kept aligned when large language models propose changes to structured knowledge.

Executive Summary

WikidataRepairEval is a research benchmark built from real Wikidata repair events. It reconstructs historical fixes, attaches a frozen contemporary graph context, classifies each case by the kind of information needed to solve it, and evaluates whether a system can produce a repair that is valid, auditable, and aligned with the historical human-maintained outcome.

The project addresses a central problem in neuro-symbolic artificial intelligence: language models can generate plausible graph updates, but they may hallucinate, overfit to local patterns, or satisfy constraints by destroying useful knowledge. Knowledge graphs, in contrast, are structured and auditable, but they are incomplete and need constant maintenance. This benchmark provides a controlled way to measure whether a proposed repair is genuinely good rather than merely syntactically acceptable.

The current release should be understood as the first minimum publishable unit of a larger Guardian-style research program. It contains the benchmark substrate, validity rules, proposal contracts, an offline evaluator, and a zero-shot reasoning-floor baseline. It does not yet implement the full future protocol in which a verifier repeatedly guides a model through multi-step repair attempts.

The Problem It Tries To Solve

Knowledge graphs need repair

Wikidata contains structured claims and constraints, but violations arise naturally as facts change, editors add data, and property constraints evolve. A repair may change an entity statement, or it may reveal that the constraint itself has become outdated.

LLMs are useful but risky

A language model can suggest plausible repairs, explain its reasoning, and generalize across domains. The same model can also invent facts, ignore provenance, or choose a shortcut that makes a violation disappear while degrading the graph.

Verification needs structure

A graph update should be treated as a transaction with a target, evidence, uncertainty, and a validity check. The benchmark is designed to test that transaction, not only the surface ability to emit a patch.

Scientific Value

The benchmark is valuable because it turns graph repair into a controlled scientific question: what information was required, what kind of repair was historically accepted, and whether a model can reproduce that behavior without violating symbolic constraints.

Scientific contribution Why it matters
Historical ecological validity The cases come from real Wikidata maintenance events rather than synthetic corruption. This makes the benchmark closer to the actual conditions under which graph repair systems would operate.
Information-necessity taxonomy Cases are separated by whether the repair can be solved by logic, local graph context, or external information. This lets researchers distinguish reasoning failures from retrieval failures.
A-box versus T-box separation The project distinguishes repairs to factual instance data from reforms to constraints or schema. Many benchmarks conflate these, but they represent different scientific problems.
Temporal validity controls Historical repairs are judged carefully against a frozen later snapshot. The benchmark avoids pretending that every past repair remains meaningful after the world and graph have changed.
Reasoning-floor baseline Zero-shot model performance is measured before retrieval, tools, memory, verifier loops, or learning interventions are added. This gives later protocol gains a defensible baseline.
Auditability as a benchmark requirement Accepted proposals are expected to include rationale, provenance, and uncertainty. This aligns evaluation with trustworthy knowledge maintenance rather than raw answer matching alone.

How The Data Is Built

The data construction process converts public Wikidata maintenance history into benchmark cases. The important idea is that a case is not just a before-and-after value. It is a repair event with provenance, context, classification, and evaluation targets.

Mine candidate repair events

The process starts from historical Wikidata constraint-violation reports. When an item disappears from a report between two report snapshots, that disappearance is treated as a candidate repair signal. The system guards against simple report reorganization by checking whether the same item moved elsewhere rather than being repaired.

Reconstruct the repair

Each candidate is traced back to the relevant Wikidata edit history. If the entity statement changed, the case becomes an A-box repair. If the property constraint changed instead, it becomes a T-box reform. The benchmark records the target, action, old value, new value, editor metadata, and repair provenance.

Check persistence

The repair is tested against the frozen contemporary graph state. Non-delete repairs must still be meaningful in the graph. Cases that no longer support the same logical evaluation question are filtered rather than silently reinterpreted.

Attach frozen world-state context

Each repair receives graph context layers: the focus node, English labels and descriptions, one-hop neighborhood information, property constraints, and optional before-and-after schema context for T-box cases. This gives systems controlled access to local graph evidence without using the current target-property value as the historical before-state.

Add popularity metadata

Each focus entity receives a popularity score derived from pageviews, graph degree, and sitelinks. This enables controlled analysis of head entities versus long-tail entities, where model familiarity may differ sharply.

Classify the benchmark case

A deterministic classifier labels A-box cases as logical, local, or external according to the information needed to solve the repair. T-box cases are labeled separately as schema reforms, with subtypes that describe the kind of constraint change when possible.

Create reproducible evaluation subsets

The full benchmark remains the complete record. For paper-scale evaluation, a deterministic manifest can select all A-box cases and cap repeated T-box cases from the same property-level schema update. This keeps the evaluation reproducible without letting repeated manifestations of one schema edit overwhelm the analysis.

Benchmark Taxonomy

Type A

Logical repairs

The rule itself or internal consistency determines the fix. Examples include format corrections, singleton allowed-value constraints, or deletion of an invalid value. These cases test reasoning without retrieval.

Type B

Local repairs

The answer is available in the focus node or immediate graph neighborhood. These cases test whether a system can use structured context rather than relying on open-ended outside knowledge.

Type C

External repairs

Local graph context is insufficient. The repair requires information outside the immediate graph view, such as a source, external database, historical fact, or web evidence.

T-box

Schema reform

The historical fix changed a property constraint rather than an entity statement. These cases capture concept drift, outdated rules, overly narrow constraints, and evolving modeling practice.

Repair tracks

Track What changed historically What it tests
A-box repair A factual statement on a focus entity changed. Can a system repair instance data while preserving graph consistency?
T-box reform A property constraint or schema rule changed. Can a system recognize when the rule is wrong or outdated rather than the entity data?

Temporal Validity

Historical repair benchmarks face a time-travel problem: an edit that was correct in the past may look wrong against a later graph snapshot because the entity, constraint, or real world changed.

The benchmark question

The benchmark does not ask whether today's graph has today's violation under today's rule. It asks whether the same logical problem still exists when the historical repair target is evaluated with frozen contemporary context.

The target-property rule

Current graph context may be used for surrounding labels, neighbors, and constraints. The edited target property is the exception: its historical pre-repair state must be reconstructed from repair metadata, not copied from the frozen world-state snapshot.

How It Is Evaluated

Evaluation is designed to answer whether a proposed repair is executable, historically aligned, semantically appropriate, and auditable. It is stricter than simply checking whether a model produced valid JSON or mentioned the correct entity.

A-box evaluation

For factual entity repairs, the evaluator reconstructs the pre-repair target-property state, applies the proposed operations in memory, and compares the resulting target property against the historical repaired state. A successful proposal should be executable against the intended entity and property, preserve relevant information, avoid supported local constraint regressions, and include rationale, provenance, and uncertainty.

Important signals include functional success, exact historical agreement, information preservation, provenance completeness, auditability completeness, regression behavior, conversion rate when multiple attempts exist, and token or cost telemetry when available.

T-box evaluation

For schema reforms, the evaluator compares the proposed post-reform constraint signature against the historical post-reform signature. Exact agreement remains strict. The benchmark also records family-level semantic compatibility so that evaluation can distinguish a completely wrong schema edit from a plausible reform that has the right direction but does not exactly match the historical signature.

Important signals include exact action match, exact signature match, semantic-family success, whether the proposal targets the changed constraint family, signature overlap, whether current values would be admitted by the reform, provenance completeness, and auditability completeness.

Track diagnosis

The benchmark also supports a diagnostic task: decide whether the correct locus of repair is A-box, T-box, or ambiguous. This matters because a model may correctly see that something is wrong but repair the wrong layer. In future protocol systems, locus selection can become part of the control flow rather than only an analytic score.

Reasoning floor

The reasoning floor is the zero-shot pre-intervention baseline. It asks what current models can do without tools, retrieval, memory, rejection sampling, or verifier-guided retries. It uses fixed prompt bundles with different amounts of context: a minimal case payload, logic and constraint context, and local graph context. Later systems should be compared against this floor so that claimed improvements can be attributed to protocol support rather than to an undefined baseline.

Zero-Shot Baseline Prompts

For the main zero-shot baseline evaluation, each benchmark case is evaluated with four model prompts: two ablation bundles multiplied by two prediction tasks.

Ablation 1

Logic-only context

The model receives the sanitized case payload plus the directly relevant constraint context. This tests whether the repair can be solved from the rule structure without broader graph neighborhood evidence.

Ablation 2

Local-graph context

The model receives the sanitized case payload plus pruned local graph context and constraint context. This tests whether one-hop graph evidence improves repair and locus decisions.

Prediction 1

Track prediction

The model predicts whether the correct repair locus is an A-box entity repair, a T-box schema reform, or an ambiguous case. This measures locus-selection ability separately from repair content.

Prediction 2

Repair proposal

The model proposes the actual repair in the appropriate normalized form. A-box proposals edit entity values; T-box proposals reform the relevant property constraint signature.

Prompt count per case: 2 ablation bundles x 2 prediction tasks = 4 prompts

Representative Examples

These examples are simplified for explanation. They illustrate the types of reasoning the benchmark tries to separate.

Type A A-box

Format correction

A plant identifier contained an extra trailing slash. The property constraint described the allowed identifier format, and the historical repair removed the invalid character. This is logical because the rule itself provides the repair signal.

What a model must do: apply the format rule without inventing a new identifier or deleting unrelated values.

Type B A-box

Local context resolves the target

A case involving a use-restriction status had multiple candidate values, and the historical repair removed one while preserving another. The relevant truth signal was available in the focus-node context rather than requiring outside retrieval.

What a model must do: read the local graph state carefully and avoid satisfying the constraint by discarding useful surviving information.

Type C A-box

External grounding needed

A historical-person case changed a value for a conviction-related property from one legal or historical concept to another. The local graph context alone is not enough to know the correct replacement with confidence.

What a model must do: seek or cite evidence beyond the local graph rather than guessing from labels.

T-box

Constraint removed after repeated violations

A genomic-coordinate property previously had a distinct-values constraint. The historical schema edit removed that constraint while preserving other constraints such as format and property scope. In this case, the repair target is the rule, not the individual entity values that appeared in the violation report.

What a model must do: recognize that the property model changed and propose an appropriate schema reform rather than editing each violating entity independently.

T-box

Constraint parameter refinement

A person-identifier property had language-related constraint metadata that was historically adjusted. This kind of case tests whether a system can reason about constraint parameters and modeling practice, not only about subject-entity facts.

What a model must do: edit the constraint signature in the right semantic direction and target the actual changed constraint family.

Main Strengths

Real repair events

The benchmark is anchored in actual Wikidata maintenance, making it more representative than synthetic corruption tasks.

Clear task decomposition

Logical, local, external, and schema-reform cases support fine-grained analysis of where systems fail.

Temporal safeguards

The benchmark explicitly separates historical repair targets from frozen contemporary context, reducing leakage and invalid comparisons.

Auditable evaluation

Proposals are judged not only by output value but also by executability, provenance, uncertainty, and local side effects.

A-box and T-box coverage

Separating instance-data repair from schema reform makes the benchmark relevant to concept drift and evolving modeling practice.

Baseline discipline

The reasoning floor provides a stable control condition for later retrieval or verifier-guided protocols.

Main Weaknesses And Risks

Dataset skew

T-box cases dominate the full benchmark. Many cases can originate from the same property-level schema edit, so naive aggregate scores may overrepresent a small number of repeated reform events. The deterministic capped subset mitigates this for paper evaluation but does not remove the underlying skew from the full historical record.

Historical repair is an imperfect gold standard

A human-maintained edit is a strong alignment proxy, but it is not proof of universal correctness. Some historical repairs may reflect local community conventions, incomplete evidence, or later-revised modeling assumptions.

Temporal complexity remains hard

The project has explicit temporal validity rules, but historical graph repair is inherently difficult. Entities may be merged, deleted, reclassified, or independently repaired. Some cases must be discarded because they no longer express a stable evaluation question.

External evidence is not yet fully operationalized

Type C cases identify the need for outside information, but the current minimum publishable unit does not yet include retrieval-backed evidence collection as a full protocol layer. This limits what can be concluded about evidence-grounded repair behavior.

Evaluation is necessarily partial

The evaluator can check supported constraint families and compare against historical targets, but it cannot fully reproduce every possible Wikidata validation rule or every community judgment. Some semantic quality remains hard to automate.

Prompt leakage must be actively avoided

Because the frozen world state may include post-repair target-property values, prompt builders must reconstruct the pre-repair target property from benchmark metadata. Incorrect use can accidentally leak the answer.

Current scope is pre-protocol

The repository establishes the benchmark, evaluator, and zero-shot control. It does not yet implement the full multi-turn Guardian protocol with verifier feedback, retrieval, and iterative repair attempts.

Future Work

The natural continuation is to build protocol layers on top of the benchmark without weakening the benchmark itself.

Live verifier runtime

Add a repair-time verifier that can reject candidate transactions and return structured diagnostics before a proposal is finalized.

Retrieval-augmented repair

Add evidence collection for Type C and evidence-heavy T-box cases, with normalized provenance that can be evaluated rather than merely displayed.

Guardian-style iterative protocols

Test bounded multi-attempt workflows where model proposals are checked, rejected, revised, and eventually accepted or abstained.

Actionable locus selection

Move A-box versus T-box diagnosis from a separate analytic task into a control decision that determines the repair route.

Cost-quality frontier analysis

Measure not only correctness but also attempt count, retrieval calls, verifier calls, tokens-to-fix, latency, and cost per accepted repair.

Broader semantic validation

Expand supported validation families and improve semantic scoring for schema reforms, especially where exact historical agreement is too narrow to capture useful behavior.

Bottom Line

WikidataRepairEval is best understood as a benchmark for trustworthy knowledge-graph repair, not just a dataset of edits. Its key contribution is the combination of real historical repair events, information-necessity labels, A-box/T-box separation, temporal validity rules, auditable proposal contracts, and a zero-shot baseline.

Its scientific value comes from making future claims about LLM-assisted graph repair measurable. A system can be asked not only whether it produced an answer, but whether it chose the right repair layer, used the right information source, preserved knowledge, supplied evidence, survived symbolic checks, and improved over a well-defined reasoning floor.

This report is intentionally standalone. It summarizes the benchmark's goals, data model, evaluation philosophy, limitations, and future research direction without requiring the reader to inspect repository internals.
